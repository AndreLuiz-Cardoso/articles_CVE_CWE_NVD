Identifying CWEs in Real Code

The most practical skill in vulnerability management is the ability to read code and identify the weakness patterns that will lead to exploitable vulnerabilities. Let's analyze a real code example.

The Code

pythonimport sqlite3

def get_user(username):
    conn = sqlite3.connect('users.db')
    cursor = conn.cursor()
    query = "SELECT * FROM users WHERE username='" + username + "';"
    cursor.execute(query)
    user = cursor.fetchone()
    conn.close()
    return user

Identified CWE: CWE-89, Improper Neutralization of Special Elements Used in an SQL Command (SQL Injection)

What is wrong: The code constructs an SQL query by concatenating user-supplied input directly into the query string. The username variable is inserted into the SQL query using string concatenation — "'" + username + "'" — without any validation, sanitization, or parameterization.

The Attack Scenario

An attacker calls this function with a crafted username:

pythonget_user("' OR '1'='1")

The resulting query becomes:

sqlSELECT * FROM users WHERE username='' OR '1'='1';

Because '1'='1' is always true, this query returns every user in the database. The attacker now has access to all user records without knowing any valid username.

More destructively, an attacker could use:

pythonget_user("'; DROP TABLE users; --")

Producing:

sqlSELECT * FROM users WHERE username=''; DROP TABLE users; --';

This deletes the entire users table.

Security Implications


Unauthorized data access: Any attacker can retrieve records for any user
Authentication bypass: If this function is used for login, an attacker can bypass authentication entirely
Data destruction: An attacker can delete or modify database contents
Data exfiltration: An attacker can extract the entire database contents


The Fix: Parameterized Queries

pythonimport sqlite3

def get_user(username):
    conn = sqlite3.connect('users.db')
    cursor = conn.cursor()
    # SECURE: parameterized query
    query = "SELECT * FROM users WHERE username = ?"
    cursor.execute(query, (username,))
    user = cursor.fetchone()
    conn.close()
    return user

The ? placeholder tells the database driver to treat the username value as pure data, never as part of the SQL command structure. No matter what the user supplies, it cannot modify the query's structure.

Additional controls:


Input validation: Even with parameterized queries, validate that usernames conform to expected format (alphanumeric, within length limits)
Least privilege: The database user used by the application should have SELECT-only access on the users table, not the ability to DROP tables
Error handling: Catch database exceptions and return generic error messages — never expose raw database errors to the user, as they reveal schema information
Logging: Log failed queries for security monitoring


This single-function example illustrates why CWE-89 consistently ranks among the most dangerous software weaknesses. The fix is straightforward and well-understood — yet SQL injection vulnerabilities continue to appear in production code because developers forget to apply parameterized queries consistently, or inherit vulnerable code from others.

The lesson: security is not about knowing the theory. It's about applying it consistently, in every function, on every code path, every time.
