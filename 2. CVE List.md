How CVE IDs Are Assigned and Who Manages the Process

The CVE system wouldn't work if anyone could create CVE entries arbitrarily or if the process were chaotic and inconsistent. Behind every CVE ID is a structured process managed by an international network of organizations.

Who Manages the CVE List?

The CVE Program is overseen by MITRE Corporation, a nonprofit organization that has managed it since its creation in 1999 under sponsorship from the US Department of Homeland Security (DHS) and the Cybersecurity and Infrastructure Security Agency (CISA).

MITRE serves as the CVE Program Root, setting the policies and standards that govern how CVE IDs are assigned and what information a CVE entry must contain.

The Role of CVE Numbering Authorities (CNAs)

The sheer volume of vulnerabilities discovered worldwide makes it impossible for any single organization to manage all CVE assignments. This is where CVE Numbering Authorities (CNAs) come in.

CNAs are organizations authorized to assign CVE IDs within their defined scope. There are currently over 200 CNAs worldwide, including:


Software vendors like Microsoft, Apple, Google, and Red Hat, who assign CVE IDs to vulnerabilities found in their own products
Bug bounty platforms like HackerOne and Bugcrowd, which assign CVE IDs to vulnerabilities reported through their programs
Research organizations and security companies that discover vulnerabilities in third-party software
National and regional CERTs (Computer Emergency Response Teams) that coordinate vulnerability disclosure in their regions


The CVE Assignment Process

When a security researcher discovers a vulnerability, the process typically works as follows:

Step 1: Discovery and Reporting. The researcher identifies a vulnerability and determines whether it's new (not already assigned a CVE ID). They contact the appropriate CNA, usually the software vendor's security team or a coordinating CNA like CERT/CC.

Step 2: CVE ID Reservation. The CNA reserves a CVE ID for the vulnerability before it is publicly disclosed. This allows the vendor time to develop and release a patch while the CVE entry is being prepared.

Step 3: Coordinated Disclosure. The vulnerability details remain confidential while the vendor develops a fix. The standard disclosure period is 90 days, popularized by Google Project Zero.

Step 4: Public Disclosure. When the patch is ready (or after 90 days), the CVE entry is published with full details: description, affected versions, severity score, and links to patches and advisories.

Step 5: NVD Enrichment. After publication, the National Vulnerability Database (NVD) adds additional analysis, CVSS scores, and mitigation guidance to the CVE entry.

This structured process balances the need for public disclosure (so organizations can protect themselves) with responsible coordination (giving vendors time to fix issues before attackers learn about them).
